# الخطة الرئيسية للنظام (لوكل + تابلت)

> وثيقة مرجعية رسمية لنظام إدارة مصنع الزي الموحد.
> النشر: **On‑Premise** — الواجهة: **كمبيوتر + تابلت (Touch Friendly)** — تتبع الدفعات: **حل غير مكلف**.

---

## 0) القرارات المحسومة (Baseline)

1. **النشر:** لوكل (On‑Premise) داخل الشركة.
2. **الاستخدام:** واجهة تشتغل على كمبيوتر + تابلت (Touch Friendly).
3. **التتبع:** حل غير مكلف بدل QR في البداية، مع إمكانية إضافة QR لاحقًا.
4. **النطاق:** كل ما يدخل النظام يجب أن يكون له: شاشة / إجراء / تقرير / سياسة إعدادات. خلاف ذلك = خارج النطاق.
5. **أسماء الأقسام:** عامة — القص / الخياطة / التطريز / الجودة / التغليف والكي / التسليم.
6. **سياسة حد أدنى 14 قطعة:** معطلة افتراضيًا — تُستخدم عند الحاجة بقرار إداري صريح فقط.

---

## 0.5) تحليل الوضع الحالي (As‑Is) + نقاط الألم

> مبني على وصف مدير الإنتاج للعمليات اليومية.

### نقاط الألم الرئيسية

| الرقم | نقطة الألم | الأثر التشغيلي |
|---|---|---|
| 1 | تسجيل الطلبات في دفاتر ورقية | ضياع الطلب، صعوبة البحث، لا تاريخ موثق |
| 2 | نقل الدفعات بين الأقسام بلا توثيق رسمي | فقدان قطع، تبادل اتهامات، لا مسؤولية واضحة |
| 3 | فروقات الكميات تُكتب كملاحظة جانبية | لا تُحسب، لا تُنسب لأحد، تتراكم بلا علاج |
| 4 | اعتماد التصاميم والتطريز شفهيًا | نزاعات مع العملاء، إعادة عمل، هدر وقت |
| 5 | طلبات الشراء شفهية أو على واتساب | تأخير، شراء خاطئ، لا سند للمراجعة |
| 6 | جدولة المشتريات عشوائية | توقف الإنتاج لانتظار القماش |
| 7 | المخزون تقديري لا فعلي | شراء زائد أو نقص مفاجئ |
| 8 | الجودة تُقيَّم شفهيًا | لا إحصاء، لا مسؤولية، لا تحسين ممكن |
| 9 | سندات التسليم ورقية | فقدان السند، نزاعات كميات مع العملاء |
| 10 | إغلاق الطلب بلا تحقق | لا أرشيف، لا إمكانية إعادة التشغيل بعد أشهر |
| 11 | إعادة الطلب تعتمد على الأشخاص | لو غاب الشخص = لا إعادة أمينة للطلب |
| 12 | العينات تتداخل مع الإنتاج الأساسي | تأخير الطلبات العادية بسبب عينة واحدة |
| 13 | تتبع الديون والآجل شخصي | خسارة مبالغ أو نسيان تحصيل |

## 1) أفضل حل لتتبع الدفعات “سهل وغير مكلف”

نهج تدريجي على ثلاث مراحل:

### 1.1 الخيار الموصى به (الأرخص والأبسط) — Phase 1
- كل دفعة لها **رقم/كود** بالصيغة: `B-YYYY-XXXXXX` (مثال: `B-2026-000123`).
- يطبع على ورقة صغيرة أو ملصق عادي ويلصق على كيس/كرتون الدفعة.
- في التابلت: شاشة “استلام/تسليم” فيها:
  - إدخال الكود يدويًا، **أو**
  - اختيار من قائمة “دفعات اليوم”.

**لماذا هذا أفضل كبداية؟**
- لا يحتاج كاميرا ولا QR ولا طابعات خاصة.
- لا يكلف شيء تقريبًا.
- سريع إذا صممنا الشاشة صح (أزرار كبيرة + بحث فوري).

### 1.2 الخيار الاقتصادي جدًا (اختياري) — Phase 2
- نطبع **Barcode 1D (Code128)** على الملصق.
- نستخدم **قارئ باركود USB** رخيص (مثل اللي يستخدم في الكاشير).
- عادة أرخص وأبسط من اعتماد كاميرات التابلت وQR.

### 1.3 QR (اختياري لاحقًا) — Phase 3
- يضاف لما تستقر العمليات ويكون الفريق جاهز.
- يحتاج: طباعة واضحة + كاميرا تابلت جيدة + ضبط إضاءة + تدريب.

### القرار النهائي
- **Phase 1:** كود دفعة + اختيار من قائمة “دفعات اليوم”.
- **Phase 2:** Barcode 1D + قارئ رخيص (اختياري).
- **Phase 3:** QR (اختياري حسب الرغبة).

## 2) الهيكل الموحد للنظام (الخماسية الأساسية)

كل شيء يدور حول 5 كيانات لمنع التشتت:

1. **الطلب Order** — كيان واحد.
2. **بنود الطلب Order Items** — لأن الطلب قد يحتوي عدة قطع/أقمشة/تصاميم.
3. **الدفعات Batches** — تشغيل بالدفعات + قص جزئي + تسليم مرحلي.
4. **حركات الانتقال والفروقات Transfers & Quantity Movements** — بدل الدفاتر.
5. **الملفات والأرشفة Files + Archive Checklist** — إعادة الطلب بدون أشخاص.

> أي رؤية/مقترح لا يرتبط بهذه الخماسية = غالبًا خارج نطاق النظام الأساسي.

---

## 3) متطلبات “لوكل + تابلت” بدون تكلفة زائدة

### 3.1 تشغيل لوكل (On‑Premise)
- سيرفر داخل الشركة (كمبيوتر قوي أو سيرفر صغير).
- قاعدة بيانات محلية.
- **نسخ احتياطي يومي تلقائي** (إلزامي).
- صلاحيات شبكة داخلية (Intranet).

### 3.2 التابلت (Touch Friendly)
- واجهات كبيرة (أزرار واضحة).
- “شاشة سريعة” لكل قسم: استلام / تسليم / تسجيل إنتاج / تسجيل جودة.
- **PIN** بدل كلمة مرور طويلة في أرض المصنع.
- “قائمة شغل اليوم” لتقليل البحث.

> **Offline على التابلت:** مؤجل لمرحلة لاحقة (يزيد التعقيد والتكلفة).

## 4) خطة التنفيذ المرحلية (Roadmap)

### Phase 0 — الحوكمة والبنية الأساسية (1–2 أسبوع)

**المحتوى:**
- المستخدمون والأدوار والصلاحيات.
- نظام PIN لأرض المصنع.
- Audit Log لكل تغيير مهم.
- إعداد السياسات التشغيلية (أقسام / سياسة 14 قطعة = معطلة افتراضيًا).

**الاعتماديات:** لا يوجد — نقطة البداية.  
**المخرج:** النظام جاهز لإدخال البيانات الأساسية.

---

### Phase 1 — MVP خط الإنتاج (4–6 أسابيع) — **يوقف الدفاتر**

**المحتوى:**
- الطلبات + البنود + المسارات (إنتاج / عينة / تعديل).
- الدفعات + تسليم/استلام + اعتماد مزدوج بـ PIN.
- تسجيل إنتاج يومي (ممنوع بدون اعتماد القسم).
- فروقات الكميات كحركات رسمية منسوبة لقسم/شخص.
- الأرشفة Checklist + رفع صور.
- تتبع الدفعات: **كود دفعة `B-YYYY-XXXXXX` + قائمة دفعات اليوم**.

**الاعتماديات:** Phase 0 مكتمل.  
**المخرج:** المصنع يمشي رقميًا في الحركة اليومية — لا دفاتر.

---

### Phase 2 — المشتريات والمخزون (4–6 أسابيع)

**المحتوى:**
- طلب شراء داخلي + اعتماد + جدولة.
- حركات مخزون + لفات قماش + خيوط تطريز.
- حد أدنى للمخزون + تنبيه.
- جرد شهري إلزامي + فروقات.
- (اختياري) Barcode 1D + قارئ USB للدفعات.

**الاعتماديات:** Phase 1 مكتمل ومستقر.  
**المخرج:** المشتريات والمخزون مرتبطان بالإنتاج — لا نقص مفاجئ.

---

### Phase 3 — التسليم والمالية التشغيلية والعينات (4–6 أسابيع)

**المحتوى:**
- سند تسليم جزئي/كامل + فاتورة مرتبطة.
- صندوق قبض/صرف.
- الآجل / التحصيل + تذكيرات تلقائية.
- نظام العينات الكامل + تقارير التحويل لمبيعات.
- تصنيف العملاء (نقدي / آجل / متعثر).

**الاعتماديات:** Phase 1 + Phase 2 مكتملان.  
**المخرج:** دورة حياة الطلب مكتملة ماليًا من الإنشاء حتى التحصيل.

---

### Phase 4 — التحليلات والكتالوج والقدرات المتقدمة

**المحتوى:**
- كتالوج سنوي للتطريز (تلقائي من الأرشيف).
- لوحات تحليل: اختناقات + إنتاجية + هدر + التزام بالمواعيد.
- (اختياري) سعة الأقسام وتوازن التحميل.
- (اختياري) QR code للدفعات بعد استقرار العمليات.

**الاعتماديات:** Phases 1-3 مكتملة + بيانات تاريخية كافية.  
**المخرج:** أدوات إدارية لدعم القرار والتحسين المستمر.

## 4.5) رؤى 2026 → متطلبات قابلة للتنفيذ

> تحويل الرؤى التشغيلية لعام 2026 إلى متطلبات ملموسة مدرجة في مراحل التنفيذ.

| رقم | رؤية 2026 | المتطلب التنفيذي | المرحلة |
|---|---|---|---|
| 1 | **التطريز:** اعتماد رسمي قبل التنفيذ | شاشة Mockup + توقيع اعتماد إلكتروني + ملف عميل (خيوط/ألوان/غرز) | Phase 1 |
| 2 | **التطريز:** فصل ماكينة العينات | تصنيف Resource مستقل لماكينة العينات في جدولة الأقسام | Phase 1 |
| 3 | **الزي الموحد:** دعم كل الأنواع | بنود الطلب تدعم: قميص/بنطال/جاكيت/بدلة/عباءة/مريلة/زي طبي/أي قطعة أخرى | Phase 1 |
| 4 | **المشتريات:** جدولة منضبطة | يومان أسبوعيًا افتراضيًا + استثناء طارئ بموافقة موثقة | Phase 2 |
| 5 | **العينات:** منع الإسراف | سقف عينات للمندوب + تقرير تحويل عينة → مبيعات | Phase 3 |
| 6 | **التحصيل:** متابعة الآجل | تصنيف العملاء (نقدي/آجل/متعثر) + تذكيرات + تقارير ديون | Phase 3 |
| 7 | **القماش الزائد:** توثيق رسمي | فروقات الكميات = حركة كمية رسمية (زيادة/تالف/هدر) منسوبة لقسم | Phase 1 |
| 8 | **الآجل:** منع تكرار الديون | منع تنفيذ طلب جديد للعميل المتعثر إلا بموافقة موثقة | Phase 3 |

## 5) “تصحيح المقترحات العشوائية” (Out‑of‑Scope)

- **شراء ماكينة / توسعة المكان / فتح مصنع:** قرارات إدارية يدعمها النظام بتقارير (اختناقات/تأخير/تكلفة أوفرتايم).
- **السيارات والحضور:** خارج نطاق نظام المصنع الأساسي (ممكن لاحقًا كنظام فرعي).
- كل ما يدخل النظام يجب أن يكون: **شاشة / إجراء / تقرير / سياسة إعدادات**.

---

## 6) تصنيف المقترحات التشغيلية

يستخدم هذا التصنيف لتوجيه فريق التطوير وتفادي الخلط بين أنواع المتطلبات.

| المقترح | التصنيف |
|---|---|
| تتبع الدفعات بكود + قائمة اليوم | ميزة نظام |
| PIN لاعتماد التسليم/الاستلام | ميزة نظام |
| Audit Log لكل تغيير مهم | ميزة نظام |
| أسماء الأقسام عامة | قرار إداري ✅ محسوم |
| سياسة 14 قطعة معطلة افتراضيًا | سياسة ✅ محسومة |
| جدولة المشتريات يومين أسبوعيًا | سياسة |
| منع إنتاج لعميل متعثر | سياسة |
| شراء ماكينات جديدة | قرار إداري — يدعمه النظام بتقارير |
| نظام الحضور والسيارات | خارج النطاق (نظام فرعي مستقل لاحقًا) |
| المحاسبة العامة وشجرة الحسابات | خارج النطاق |
| تطبيق موبايل للتسليم خارج المصنع | خارج النطاق — Phase 4 اختياري |
